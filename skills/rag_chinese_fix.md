# Skill: Chinese-Aware RAG Optimization

## 1. 核心目標
解決中文環境中，因缺乏空格分詞而導致的 RAG (檢索增強生成) 匹配失效問題，確保系統能精確定位「知識圖譜實體」與「原始郵件」。

## 2. 實作規範 (Implementation Rules)

### A. 實體匹配 (Entity Matching)
- **錯誤作法**：使用 `query.split()` 後匹配檔案名。
- **正確作法**：使用「反向包含檢索」。
  - 檢查 `實體名稱.lower()` 是否包含在 `使用者提問.lower()` 中。
  - 範例：若檔名為 `東林易.md`，提問為 `請分析東林易的案件`，則應判定匹配。

### B. 資料庫檢索 (Database Retrieval)
- **多重關鍵字策略**：
  1. 先呼叫 AI (`extract_keywords`) 提取 3-5 個獨立的關鍵字。
  2. 使用 `OR` 邏輯組合多個 `LIKE %keyword%` 的 SQL 條件。
  3. 必須保留原始提問字串作為最後一個搜尋項。

### C. 內容截斷與 Context 組合
- 知識圖譜實體優先於原始郵件。
- 郵件內容應截斷至 400-600 字，以節省 Token 並避免噪音。

## 3. 測試清單 (Test Cases)
- [ ] 測試長句匹配（無空格）：`查詢2025年東林易的異常` -> 應匹配 `東林易.md`
- [ ] 測試公司全名匹配：`東林易塑膠有限公司` -> 應匹配 `東林易.md`
- [ ] 測試多重關鍵字：`分析張小姐與專案A的進度` -> 應同時提取並搜尋「張小姐」與「專案A」
