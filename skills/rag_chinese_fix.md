# Skill: Chinese-Aware RAG Optimization (中文化檢索優化)

## 1. 核心需求
針對繁體中文工業環境，解決常見的「分詞斷開」與「特有名稱匹配失敗」問題。

## 2. 實作規範 (Implementation Rules)

### A. 實體精準匹配 (Entity Matching)
- **策略**: 全專案強制使用 `utf-8-sig` (BOM) 並使用 `safe_read_file` 函數進行多重回退讀取。
- **演化教訓**: 
    - 寫入 JSON 或 Markdown 時必須指定 `encoding='utf-8-sig'`，否則不同工具間會出現解碼衝突。
    - 原始數據恢復應優先從 `emails.db` 進行原子重建，而非依賴不穩定的檔案緩存。
    - **技術 ID 歸併 (v1.3)**: 料號（如 R1-1000 vs R1-10000）可能存在誤打 (Typo)，系統應對編輯距離 < 2 的技術代碼進行模糊預警或別名合併。
- **正確模式**: 使用「全字匹配」或「正則表達式」。
  - 檢查 `EntityName.lower()` 是否包含在 `Query.lower()` 中。
  - 範例：若 Query 為 "請查詢張三的報告"，系統應自動定位到 `張三.md`。

### B. 數據庫檢索優化 (Database Retrieval)
- **關鍵字提取**: 
  1. 調用 AI 提取 3-5 個工業核心關鍵字。
  2. 使用 `LIKE %keyword%` 進行 SQL 查詢。
  3. 權重分配：標題匹配 > 內文匹配。

### C. 上下文拼接 (Context Assembly)
- 確保拼接後的 Context 包含實體的原始維度路徑（例如：HR/人員）。
- 單次檢索窗口限制在 1500 字以內，避免 Token 溢出或幻覺。

## 3. 測試清單 (Test Cases)
- [ ] 測試含有年份的查詢：`2025年異常報告` -> 應自動過濾 2025 年數據。
- [ ] 測試含有括號的名稱：`料號 (A區)` -> 應正確轉義正則符號。
- [ ] 測試多重維度檢索：`張三 氣泡 改善` -> 應返回張三處理氣泡異常的相關郵件。
